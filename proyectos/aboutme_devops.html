El Desafío

Durante un proceso de entrevista para una posición DevOps, me asignaron un proyecto técnico con un plazo de 1 semana: transformar una aplicación existente en un proyecto DevOps completo utilizando Docker, Kubernetes, Prometheus y Grafana.

Tomé una landing page personal simple que ya tenía (HTML, CSS, JavaScript puro) y la transformé completamente, agregando containerización, orquestación, monitoreo y simulación de fallos. El resultado fue un proyecto que debía presentar y defender técnicamente ante dos líderes técnicos del equipo.

El Reto Real

Tiempo: 7 días para aprender e implementar
Conocimiento previo: AWS Cloud Practitioner, backend/frontend básico
Stack nuevo: Docker, Kubernetes, Helm, Prometheus, Grafana
Presentación: Demo en vivo ante 2 líderes técnicos
Método: Documentación oficial + IA como asistente de aprendizaje

La Transformación

Aplicación Original

Landing page simple con HTML5, CSS3 y JavaScript vanilla. Tenía un formulario que generaba tarjetas dinámicas con información del usuario. Sin servidor, sin backend, solo archivos estáticos servidos directamente.

Servidor Express

Creé un servidor con Node.js + Express para servir la aplicación y agregar endpoints de health checks, simulación de fallos (/demo/fail) y generación de carga CPU (/demo/stress) para demostrar observabilidad.

Containerización

Dockericé la aplicación creando un Dockerfile multi-stage optimizado, imagen basada en Node Alpine para mantenerla ligera, y configuré variables de entorno para diferentes ambientes.

Orquestación Kubernetes

Implementé Kubernetes con minikube localmente, creé deployments, services, health probes (readiness/liveness), resource limits, y configuré auto-recovery para demostrar resiliencia.

Monitoreo con Prometheus

Instalé Prometheus vía Helm, configuré ServiceMonitors para scraping de métricas, y agregué métricas personalizadas de la aplicación (requests, errores, latencia, uso de CPU/memoria).

Visualización con Grafana

Configuré Grafana con dashboards para visualizar métricas en tiempo real, alertas cuando los pods se reinician, y gráficos de uso de recursos para demostrar el comportamiento bajo carga.

Proceso de Aprendizaje

El Desafío de Aprender Rápido

Tenía conocimientos básicos de cloud (AWS Cloud Practitioner) y desarrollo backend, pero nunca había trabajado con Kubernetes, Prometheus o Grafana. Tenía 7 días para aprender, implementar y estar listo para defender el proyecto técnicamente.

Mi Estrategia

Combiné documentación oficial (Kubernetes docs, Prometheus guides) con IA como asistente de aprendizaje para entender conceptos, debuggear errores y optimizar configuraciones. No copié código ciegamente - usé IA para explicarme conceptos que no entendía y validar mis implementaciones.

Lo que Aprendí en 1 Semana

  • Docker: Cómo containerizar aplicaciones, crear Dockerfiles optimizados, gestionar imágenes y volúmenes
  • Kubernetes: Conceptos de pods, deployments, services, health probes, resource limits, auto-recovery
  • Helm: Package manager para Kubernetes, creación de charts parametrizables y reutilizables
  • Prometheus: Arquitectura de monitoreo, scraping de métricas, PromQL básico, configuración de targets
  • Grafana: Creación de dashboards, paneles, queries, alertas y visualización de datos de Prometheus
  • DevOps mindset: Observabilidad, resiliencia, automatización, infrastructure as code

Implementaciones Clave

Dockerfile Optimizado

  • Multi-stage build para reducir tamaño de imagen final
  • Base image: node:18-alpine (ligera y segura)
  • Copia solo archivos necesarios (no node_modules innecesarios)
  • Usuario no-root para seguridad
  • Health check integrado en el contenedor

Kubernetes Deployment

  • Replicas configurables vía Helm values
  • Readiness probe en /health para traffic routing
  • Liveness probe para auto-recovery
  • Resource requests y limits para scheduling eficiente
  • Rolling updates con maxSurge y maxUnavailable

Endpoints de Demo

  • GET /health: Health check que Kubernetes consume
  • GET /demo/fail: Simula crash del servidor (exit code 1) para ver auto-recovery
  • GET /demo/stress: Genera carga CPU por 10s para observar métricas de uso
  • Todos los endpoints loggeados para debugging

Monitoreo Implementado

  • Prometheus scraping cada 30 segundos
  • Métricas de sistema: CPU, memoria, network I/O
  • Métricas de app: request count, error rate, response time
  • Dashboards en Grafana: Cluster overview, Pod metrics, App performance
  • Alertas configuradas para pod restarts y alta latencia
